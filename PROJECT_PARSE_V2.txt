All tables are in the public Supabase schema with Row-Level Security enabled.
All tables have created_at and updated_at (auto-updated via set_updated_at trigger).

────────────────────────────────────────────────────────────────────────────────
CORE TABLES (migration 0–3)
────────────────────────────────────────────────────────────────────────────────

TABLE profiles
PRIMARY_KEY id uuid
FK id → auth.users(id) ON DELETE CASCADE
COLUMNS display_name text, subscription_tier text DEFAULT 'free'
         CHECK (subscription_tier IN ('free','premium'))
NOTE subscription_tier added in migration 6.
RLS Users can SELECT/INSERT/UPDATE/DELETE own row (id = auth.uid()).

TABLE cards
PRIMARY_KEY id uuid (gen_random_uuid())
FOREIGN_KEY profile_id → profiles.id ON DELETE CASCADE
COLUMNS label text, tts_text text NULLABLE, image_url text NULLABLE,
        category text DEFAULT '', position integer DEFAULT 0,
        deleted_at timestamptz NULLABLE (soft delete)
NOTE tts_text and position added in migration 2. category values: "aac" | "task".
RLS Users can SELECT/INSERT/UPDATE/DELETE own cards (profile_id = auth.uid()).

TABLE schedules
PRIMARY_KEY id uuid (gen_random_uuid())
FOREIGN_KEY profile_id → profiles.id ON DELETE CASCADE
COLUMNS title text, deleted_at timestamptz NULLABLE (soft delete)
RLS Users can SELECT/INSERT/UPDATE/DELETE own rows (profile_id = auth.uid()).

TABLE schedule_items
PRIMARY_KEY id uuid (gen_random_uuid())
FOREIGN_KEY schedule_id → schedules.id ON DELETE CASCADE
FOREIGN_KEY card_id → cards.id ON DELETE CASCADE
COLUMNS position integer DEFAULT 0, is_complete boolean DEFAULT false
RLS Joined via schedules: allowed if schedules.profile_id = auth.uid().

TABLE rewards
PRIMARY_KEY id uuid (gen_random_uuid())
FOREIGN_KEY profile_id → profiles.id ON DELETE CASCADE
COLUMNS stars integer DEFAULT 0, reason text DEFAULT ''
NOTE reason is used as idempotency key in addRewardIfNew().
RLS Users can SELECT/INSERT/UPDATE/DELETE own rows (profile_id = auth.uid()).

TABLE settings
PRIMARY_KEY id uuid (gen_random_uuid())
FOREIGN_KEY profile_id → profiles.id ON DELETE CASCADE (UNIQUE constraint)
COLUMNS calm_mode boolean DEFAULT false, big_button_mode boolean DEFAULT false,
        parent_lock_enabled boolean DEFAULT true,
        grid_size integer DEFAULT 3 CHECK (grid_size IN (2,3,4)),
        sound_enabled boolean DEFAULT true
NOTE grid_size and sound_enabled added in migration 1.
RLS Users can SELECT/INSERT/UPDATE/DELETE own row (profile_id = auth.uid()).

STEP 9 — RUNTIME CONSTRAINTS
════════════════════════════════════════════════════════════════════════════════

• STATIC EXPORT ONLY
  Next.js output:"export" is set in next.config.ts. Produces static HTML/JS/CSS
  in the /out directory. No Node.js server at runtime.

• NO API ROUTES
  There are no Next.js API routes (/api/*). All server communication goes through
  the Supabase JS client directly.

• NO SERVER-SIDE RENDERING
  All pages are "use client". Data fetching is 100% client-side via Supabase JS +
  TanStack Query. No getServerSideProps, no server components with data fetching.

• SUPABASE IS THE ONLY BACKEND
  Authentication, database, and storage all go through a single Supabase project.
  No other external services at runtime (PostHog is analytics-only, opt-in).

• SUPABASE CLIENT MAY BE NULL
  supabaseClient.ts returns null when env vars are missing (e.g. CI builds).
  Every consumer must guard against null before calling Supabase. Failure to do so
  throws "Supabase client is not initialized".

• BUILD COMMAND IS --webpack
  npm run build = "next build --webpack". The --webpack flag is required because
  @ducanh2912/next-pwa is not compatible with Turbopack builds.
  (next dev uses Turbopack; next build must use webpack.)

• OFFLINE FUNCTIONALITY MUST NOT REGRESS
  All mutations must remain offline-capable. New write helpers must call
  withOfflineQueue(). Direct client().from(...).update() calls without queuing
  will break offline mode.

• AAC MUST ALWAYS REMAIN FREE
  /kid/talk page has no subscription check and never will. The UpgradeBanner
  component must never be rendered on AAC pages. This is a product commitment,
  not just a code preference.

• KID PAGES MUST NEVER DISPLAY PAYWALLS
  Kid-facing pages show UpgradeBanner only where the parent has control (e.g.
  subject/module/lesson study pages behind the parent study planner context).
  Direct hard blocks on /kid/talk, /kid/schedule, /kid/rewards, /kid/break are
  forbidden.

• POSTHOG ANALYTICS IS PRODUCTION-ONLY AND OPT-OUT
  initAnalytics() returns early unless NODE_ENV === 'production' AND a valid
  NEXT_PUBLIC_POSTHOG_KEY is set. Opt-out preference is stored in localStorage
  ("routinenest_analytics_enabled"). Session replay is explicitly disabled.
  aac_card_tapped events track category only — never label or tts_text.

• CURRICULUM CONTENT IS CLIENT-SIDE STATIC
  studySeedData.ts is a pure static TypeScript file. No DB read is needed to
  render the curriculum. The curriculum DB tables (year_categories, class_levels,
  subject_areas, study_modules, lessons, activities) are schema-present but the
  app does NOT query them — it reads studySeedData.ts directly.

• TANSTACK QUERY CACHE IS PERSISTED TO INDEXEDDB
  The full dehydrated query cache is stored in IndexedDB key
  "routinenest_query_cache". gcTime is 24 hours, staleTime is 60 seconds by
  default. This enables offline reads of any previously fetched data.

• REACT 19 STRICT HOOK RULES
  ESLint enforces react-hooks/set-state-in-effect. setState cannot be called
  directly in useEffect bodies — must be wrapped in setTimeout(..., 0) or placed
  in event callbacks.

• PROFILE ID IS STORED IN LOCALSTORAGE
  The active child profile ID is stored at "routinenest_active_profile_id" via
  profileStore.ts. This is separate from the Supabase auth user ID. The auth
  user (parent) owns profiles; the active profile is the currently selected child.

• PARENT GATE SESSION TIMEOUT IS 10 MINUTES
  Once the parent gate is unlocked via the 3-second hold, it stays unlocked for
  10 minutes. The expiry timestamp is stored in localStorage at
  "routinenest_parent_unlocked_until".

• CAPACITOR WEBDIR IS "out"
  The Capacitor native shells serve the Next.js static export from the /out
  directory. After changing web content: run `npm run cap:build` to rebuild and
  copy assets to native.

STEP 10 — FINAL PROJECT STATE SUMMARY
════════════════════════════════════════════════════════════════════════════════

OVERALL COMPLETENESS
  The core product is functionally complete for an initial launch. All five main
  user-facing features (AAC, Schedule, Rewards, Study, Break) have working
  implementations. The subscription system, onboarding, parent management tools,
  and offline support are all implemented and tested at code level.

PRODUCTION READINESS
  NOT YET PRODUCTION-READY due to one critical gap: payment integration.
  The subscription tier can be set directly in the database but there is no
  payment provider (Stripe, RevenueCat, or App Store billing) integrated.
  All other systems are functionally production-ready.

LARGEST MISSING SYSTEMS
  1. Payment Integration — Stripe/RevenueCat/App Store billing. No revenue is
     collectible without this. Tier selector UI exists but calls DB directly.
  2. Activity Gameplay Depth — 6 of 9 activity types fall back to a "Mark Done"
     button. listen_choose, speak_tap_aac, trace, routine_checkoff, parent_guided,
     and printable have no interactive components.
  3. Analytics Depth — child_progress data is captured but the /parent/study/progress
     dashboard is basic. No charts, no trends over time, no CSV export.

HIGHEST RISK AREAS
  1. Payment launch — wiring any payment provider requires careful handling of
     webhook security, subscription state reconciliation, and App Store review
     compliance for in-app purchases. High complexity, zero code present.
  2. Curriculum scale — studySeedData.ts is a single 771-line file with 126 entries.
     Adding significantly more content without a CMS or admin panel will become
     unmaintainable. The DB tables exist but are never queried; the gap between
     schema and usage is a future debt item.
  3. Offline sync edge cases — last-write-wins is a reasonable simple strategy
     but can cause data loss if the same record is modified offline on two devices
     between syncs. Deterministic resolution is not guaranteed for concurrent edits.
  4. Auth session management — profiles.id is the Supabase auth user ID (UUID from
     auth.users). A single authenticated user (parent) owns all profiles (children).
     There is no multi-parent / shared household model — adding one would require
     significant schema changes.
  5. speak.ts deprecation — the old stub file still exists and could be imported
     by mistake. tts.ts is the correct module. The stub should be removed in a
     cleanup pass.
